### 7. Debugging alertas — 3 bugs encadenados corregidos

**Bug A** — Alerta "NIVEL DE ACEITE BAJO" nunca disparaba:
- Root cause: `condition.field = "sensor_de_nivel"` pero `can_data` usa `avl_150`
- Fix: UPDATE directo en BD + NOTIFY hot-reload → alerta disparó en 23 segundos

**Bug B** — Nuevas reglas tampoco dispararían (sistémico):
- Root cause: `ConditionBuilder` usaba `sensor.key` como field, pero el rules-engine busca en `can_data` por `avl_{avl_id}`
- Fix: función `sensorFieldKey(s)` → devuelve `avl_${s.avl_id}` cuando existe, si no `s.key`

**Bug C** — Error 500 al ver alertas tras primer disparo:
- Root cause: rules-engine hacía `json.dumps(trigger_value)` antes de pasarlo a asyncpg — columna JSONB recibía un string JSON en lugar de un objeto → Pydantic `ValidationError: dict_type`
- Fix: `$5::jsonb` cast en el INSERT + reparación del registro ya almacenado con `(trigger_value #>> '{}')::jsonb`

## Notas técnicas

- `sensor.key` ≠ clave en `can_data`: el ingest-svc almacena por `avl_{avl_id}`, no por el `key` del sensor_schema. Los dos sistemas deben usar `avl_xxx` para coincidir.
- El rules-engine hot-reload funciona vía `NOTIFY rules_changed` en PostgreSQL — no requiere restart para cambios de reglas.
- Sidebar expandible implementada y revertida por decisión de producto: se prefiere la navegación horizontal compacta (TopNav).
- `tsconfig.app.tsbuildinfo` nunca se versiona (artefacto de compilación).
